Introducción

¿Qué es colorear un grafo?

La idea en simple ▸ hover para más

Explicación extendida
Un grafo G=(V,E) es una estructura matemática con un conjunto V de vértices (puntos) y un conjunto E de aristas (pares de vértices conectados). Un coloreo válido es una función f: V → {1,2,...,k} tal que para toda arista (u,v)∈E se cumple f(u) ≠ f(v). El algoritmo greedy construye esta función de forma incremental, procesando los vértices uno a uno y asignando siempre el menor número disponible.

Términos clave ▸ hover para más

  • G=(V,E) — Grafo: V son los vértices, E las aristas
  • χ(G) — Número cromático: mínimo de colores necesarios
  • Δ(G) — Grado máximo: el vértice con más vecinos
  • Greedy — Asigna el menor color disponible, sin mirar hacia adelante
¿Por qué Δ(G)+1 es la cota?
Si un vértice tiene a lo sumo Δ(G) vecinos, esos vecinos pueden bloquear como máximo Δ(G) colores distintos. El color Δ(G)+1 siempre queda libre. Por eso el greedy NUNCA necesita más de Δ(G)+1 colores, sin importar el grafo.

Garantía que demuestra el algoritmo:

χ(G) ≤ Δ(G) + 1

Nunca necesitamos más colores que (mayor cantidad de vecinos) + 1

Ejemplo visual: Δ(G)=3 → usamos 3 colores ≤ 4

Algoritmo Oficial

El Código del Algoritmo

// Precondition P: G=(V,E) simple undirected graph, V={v1..vn}, V≠∅
color[·] ← 0
colors-used ← 0
available[·] ← true

// Invariant I0: colors form range {1,...,colors-used} — no gaps
// Invariant I1: ∀j<i, color[vj] valid — no conflicts with prior vertices
// Invariant I2: colors-used ≤ Δ(G)+1
// Invariant I3: adjacent vertices have different colors

for i ← 1 to n do
  for c ← 1 to colors-used do
    available[c] ← true
  end for

  for each neighbor vj of vi with j<i do
    if color[vj] ≠ 0 then
      available[color[vj]] ← false
    end if
  end for

  c ← 1
  while c ≤ colors-used ∧ ¬available[c] do
    c ← c + 1
  end while

  if c > colors-used then
    colors-used ← colors-used + 1
    c ← colors-used
  end if

  color[vi] ← c
end for
// Postcondition Q: ∀(u,v)∈E: color[u]≠color[v], colors-used ≤ Δ(G)+1
▶ ¿Qué hace el bucle principal? (for i)
Detalle del bucle externo
El bucle for i ← 1 to n garantiza que cada vértice se procesa exactamente una vez. El orden de procesamiento es fijo: v₁, v₂, ..., vₙ. Este orden influye en el resultado final (distintos órdenes pueden producir distintas cantidades de colores), pero la cota Δ(G)+1 siempre se mantiene independientemente del orden.
▶ ¿Qué hace el reinicio? (for c)
¿Por qué reiniciar hasta colors-used?
El invariante I0 garantiza que los colores forman la secuencia {1,...,colors-used} sin huecos. Por tanto, reiniciar solo ese rango es suficiente — no hay colores activos fuera de ese rango. Si hubiera huecos, el reinicio parcial dejaría colores "fantasma" con valores residuales.
▶ ¿Qué hace el bloqueo de vecinos?
Solo vecinos con j<i
El algoritmo solo revisa vecinos procesados antes (j < i). Los vecinos futuros (j > i) aún no tienen color asignado, así que no generan conflicto ahora. Cuando llegue su turno, ELLOS revisarán a vᵢ como vecino anterior y evitarán su color.
▶ ¿Qué hace la búsqueda y asignación?
¿Por qué c no puede superar Δ(G)+1?
Si todos los colores {1,...,colors-used} están bloqueados, cada uno está asociado a un vecino distinto con j<i. Hay a lo sumo Δ(G) vecinos → colors-used ≤ Δ(G). Al crear el nuevo color, colors-used ← colors-used+1 ≤ Δ(G)+1. El invariante I2 se mantiene siempre.

El bucle "for each neighbor vj with j<i" busca vecinos con índice menor. En i=1, no existe ningún j con j < 1, así que el bucle no ejecuta ninguna iteración. available[] queda todo true, y c=1 siempre estará disponible.

// Reset available[1]=true. Neighbor v₁(c=1): available[1]=false.
c=1→blocked, c=2 > cu(1) → cu=2; color[v₂] ← 2 🟢
El while y el if en este caso
El while arranca con c=1. Como available[1]=false (bloqueado por v₁), c se incrementa a 2. Ahora c=2 > cu=1, así que el while termina. Luego el if detecta c > cu y crea el nuevo color: cu←2, c←2. Se asigna color[v₂]=2.

i=3 → v₃ (vecinos: v₁, v₂)  ▸ hover

// Reset available[1,2]=true.
// v₁(c=1)→avail[1]=false; v₂(c=2)→avail[2]=false
c=1→blocked, c=2→blocked, c=3 > cu(2) → cu=3; color[v₃] ← 3 🟡
¿Por qué es el "peor caso"?
v₃ está conectado a v₁ (color 1) y v₂ (color 2). Los dos únicos colores existentes están bloqueados. El algoritmo no tiene más opción que crear un tercer color. Esto refleja que v₃ forma un triángulo con v₁ y v₂ (K₃), lo que matemáticamente requiere 3 colores mínimo.

Estado: cu=3, coloreados: v₁(1), v₂(2), v₃(3). Invariante I3 verificado: v₁↔v₂ (1≠2✓), v₁↔v₃ (1≠3✓), v₂↔v₃ (2≠3✓). Pendientes: v₄, v₅.

¿Por qué este grafo es interesante? ▸ hover

¿Qué es una 3-clique?
Una clique es un subconjunto de vértices donde todos están conectados entre sí. Una 3-clique (K₃) tiene 3 vértices y 3 aristas formando un triángulo. Por definición, una k-clique requiere al menos k colores: cada vértice es vecino de todos los demás en la clique. Por eso el triángulo f-g-h fuerza χ(G) ≥ 3.

Estructura del grafo

  • • 8 vértices: a, b, c, d, e (pentágono) + f, g, h (triángulo)
  • • 11 aristas: 5 del pentágono + 3 del triángulo + 3 cruzadas (a-f, b-g, d-h)
  • • Grado máximo Δ(G) = 3 (a y b tienen grado 3)
  • • Coloreo óptimo: 3 colores (el greedy lo alcanza en este caso)

Resultado ¿Por qué 3 y no 4? ▸ hover

Teorema de Brooks (1941)
Para cualquier grafo G que NO sea un grafo completo Kₙ ni un ciclo impar, se cumple χ(G) ≤ Δ(G). Este grafo (8 vértices, no completo, pentágono par) cumple la condición de Brooks y por eso el greedy puede lograr 3 colores = Δ(G), que es óptimo aquí.

Verificación Formal

La Terna de Hoare: {P} S {Q}

La lógica de Hoare explicada
La notación {P} S {Q} fue propuesta por Tony Hoare en 1969. P es la precondición (lo que se asume antes de ejecutar S), Q es la postcondición (lo que se garantiza si S termina). Los invariantes de bucle son el "puente" que conecta P con Q a través de las iteraciones: si se cumplen antes, durante y después de cada iteración, y el bucle termina, entonces Q se cumple al final.

Precondición P
(qué debe ser cierto ANTES)

G=(V,E) grafo no dirigido simple

V = {v₁,...,vₙ}, V ≠ ∅

Programa S
El algoritmo greedy

Inicialización + bucle principal de coloreo

Postcondición Q
(qué se garantiza DESPUÉS)

∀(u,v)∈E: color[u] ≠ color[v]

colors-used ≤ Δ(G)+1

I0 — Sin huecos en los colores ▸
Por qué importa I0
Si existiera un hueco (e.g., color 3 asignado pero no el 2), el reinicio parcial dejaría available[3] con valor residual de la iteración anterior, lo que haría que el algoritmo creyera erróneamente que ese color está bloqueado o libre cuando no lo está.
I1 — Coloreo previo válido ▸
I1 implica Q al terminar
Cuando el bucle termina, i=n+1, lo que significa que I1 aplica a todos los vértices v₁...vₙ. Todos están coloreados correctamente. Combinado con I3, esto es exactamente la postcondición Q.
I2 — Cota de colores garantizada ▸
Demostración de I2
Al crear un nuevo color, todos los colores actuales {1,...,cu} están bloqueados, cada uno por un vecino distinto. Hay a lo sumo Δ(G) vecinos → cu ≤ Δ(G). Después del incremento: cu+1 ≤ Δ(G)+1.
I3 — Adyacentes con colores distintos ▸
Cómo se mantiene I3
Al asignar color[vᵢ]=c, sabemos que available[c]=true, lo que significa que ningún vecino vⱼ (j<i) tiene color c. Por tanto, ninguna arista (vᵢ,vⱼ) con j<i genera conflicto. Los vecinos con j>i aún no están coloreados.
Verificación Formal · Demostración

¿Los invariantes se mantienen? (Paso inductivo)

¿Por qué inducción?
No podemos verificar manualmente cada posible grafo — hay infinitos. La inducción matemática nos permite demostrar una propiedad para todos los casos de una vez: si funciona para la base (antes de i=1) y si funciona para el paso i=k implica que funciona para i=k+1, entonces funciona para TODOS los valores de i.

Base: antes de i=1

Antes del primer vértice: color[·]=0, cu=0, available[·]=true. Todos los invariantes se cumplen trivialmente: ningún vértice está coloreado, no hay conflictos posibles, 0 ≤ Δ(G)+1.

Caso 1: reutilizamos un color (c ≤ cu) ▸

  • • available[c]=true → ningún vecino usa c → I3 se preserva
  • • cu no cambia → I2 sigue válido
  • • El color c está dentro del rango → I0 e I1 se mantienen
¿Por qué available[c]=true garantiza I3?
Si available[c]=true, significa que el bucle "for each neighbor vj" no marcó ese color como false. Eso ocurre solo cuando ningún vecino vⱼ (j<i) tiene color[vⱼ]=c. Así, al asignar color[vᵢ]=c, ninguna arista (vᵢ,vⱼ) viola I3.

Caso 2: creamos un color nuevo (c > cu) ▸

  • • Todos los colores {1,...,cu} bloqueados → vₖ tiene ≥ cu vecinos distintos
  • • vₖ tiene a lo sumo Δ(G) vecinos → cu < Δ(G)
  • • Nuevo cu = old_cu+1 ≤ Δ(G)+1 → I2 preservado
  • • Nuevo color es único → I3 y I0 preservados
¿Por qué cu < Δ(G) en ese momento?
Si todos los cu colores actuales están bloqueados, cada uno lo está por un vecino distinto de vₖ (un vecino no puede bloquear dos colores distintos). Entonces vₖ tiene al menos cu vecinos procesados → cu ≤ grado(vₖ) ≤ Δ(G). Por tanto cu ≤ Δ(G), y cu+1 ≤ Δ(G)+1.

Clave de la demostración (I2)

// At vertex vₖ — worst case analysis: neighbors_before(vₖ) ≤ Δ(G) // by definition of Δ colors_blocked ≤ neighbors_before(vₖ) ≤ Δ(G)
// So in range {1,...,Δ(G)+1}: // AT LEAST 1 slot is ALWAYS free! ∴ c ≤ Δ(G)+1 ∴ colors-used ≤ Δ(G)+1   // I2 maintained ✓

Conclusión de la inducción

Después de procesar vₖ, los 4 invariantes siguen siendo verdaderos. Por inducción matemática, se mantienen para toda la ejecución. Cuando el bucle termina (i = n+1), I1 e I3 directamente implican la postcondición Q.

Verificación Formal · Terminación

¿Por qué el algoritmo siempre termina?

¿Por qué necesitamos esto?
Demostrar que un algoritmo termina no es trivial — hay algoritmos que se ejecutan para siempre (bucles infinitos) cuando no se diseñan correctamente. La función de cota es la herramienta formal que garantiza terminación: si existe una función que siempre es positiva y siempre decrece, el bucle no puede ejecutarse infinitamente.

🔢

Bucle externo (for i=1 to n)

Cota: n − i + 1

¿Por qué n es finito?
La precondición P requiere que V={v₁,...,vₙ} con n conocido y finito. Sin esta condición, el bucle podría no tener cota superior y ejecutarse infinitamente.

🔄

Bucle de reinicio (for c=1 to cu)

Cota: cu − c + 1

Interacción con I2
El bucle de reinicio recorre c de 1 hasta colors-used. Por I2, colors-used ≤ Δ(G)+1. Entonces este bucle ejecuta a lo sumo Δ(G)+1 veces por iteración del bucle externo — siempre finito.

🔍

Búsqueda de color (while c ≤ cu)

Cota: cu − c + 2

¿Por qué siempre hay un color disponible?
El rango {1,...,Δ(G)+1} tiene Δ(G)+1 posiciones. Un vértice puede bloquear a lo sumo Δ(G) de ellas (un bloqueo por vecino, a lo sumo Δ(G) vecinos). Por lo tanto, al menos 1 posición siempre queda libre. El while siempre encuentra su condición de parada.

Terminación global

Los tres bucles son finitos y acotados. El algoritmo termina en tiempo O(n · (Δ(G)+1)) → O(n + m) con lista de adyacencia.

¿Y si el grafo fuera infinito? ▸

La precondición como guardiana de la terminación
La precondición P no solo define "qué tipo de grafo acepta el algoritmo" — también es lo que garantiza que el algoritmo termine. Si P no requiriera V finito, el bucle for tendría cota n = ∞ y la función de cota n-i+1 nunca llegaría a 0.
Reflexión · Límite Superior y Teorema de Brooks

χ(G) = Δ(G)+1 vs. χ(G) < Δ(G)+1

Casos donde χ(G) = Δ(G)+1 (el peor caso) ▸

  • 🔺 Grafo completo Kₙ: todos conectados con todos. Necesita n colores = Δ+1
  • 🔺 Ciclos impares C₃,C₅,C₇: Δ=2, pero χ=3 = Δ+1
¿Por qué los ciclos impares necesitan Δ+1?
En un ciclo impar Cₙ (n impar), al intentar 2-colorear, el "último" vértice del ciclo siempre choca con el color del primer vértice. Esto se debe a la paridad impar. Con 3 colores siempre es posible. Ejemplo: C₅ (pentágono) necesita exactamente 3 colores aunque Δ=2.

Casos donde χ(G) < Δ(G)+1 ▸

  • ✅ Grafo bipartito: siempre χ=2, sin importar Δ
  • ✅ Árbol: χ=2 (caso especial de bipartito)
  • ✅ Ciclo par C₄,C₆: χ=2, Δ=2 → χ < Δ+1
  • ✅ Nuestro Ejemplo 2: χ=3, Δ=3 → χ = Δ < Δ+1
¿Qué es un grafo bipartito?
Un grafo donde los vértices se dividen en dos grupos A y B tal que todas las aristas van de A a B (ninguna dentro del mismo grupo). Siempre se puede 2-colorear: todos los vértices de A toman el color 1 y todos los de B el color 2. Los compiladores modelan el conflicto de variables como grafos que suelen ser cercanos a bipartitos.

Teorema de Brooks (1941) ▸ hover

χ(G) ≤ Δ(G)

excepto Kₙ o ciclos impares

Relación entre Brooks e I2
El Teorema de Brooks es más fuerte que I2: dice que en casi todos los grafos χ(G) ≤ Δ(G). El invariante I2 solo garantiza ≤ Δ(G)+1 porque el greedy no siempre es óptimo. Para alcanzar el óptimo de Brooks se necesitan algoritmos más sofisticados (no greedy simple).

Limitación del greedy

El greedy no siempre es óptimo. El resultado depende del orden de procesamiento. Ejemplo: un orden "malo" puede usar más colores de los necesarios.

Encontrar el coloreo óptimo (χ(G) exacto) es NP-completo: no se conoce algoritmo eficiente para todos los casos. Por eso el greedy, aunque no siempre óptimo, es muy útil en la práctica.

Reflexión · Aplicaciones y Conclusiones

¿Para qué sirve en la vida real?

📡

Frecuencias de radio

Cómo se modela
Cada antena es un vértice. Dos antenas se conectan con una arista si su rango se superpone. El número cromático del grafo = número mínimo de frecuencias necesarias para que no haya interferencia.

💻

Compiladores (registros)

Register allocation
Cada variable es un vértice. Dos variables se conectan si sus "vidas" (intervalos de uso) se superponen. Los colores son los registros del procesador. Si se necesitan más colores que registros disponibles, el compilador "derrama" variables a memoria RAM.

📅

Horarios de exámenes

Generalización
Este problema es un caso del "graph coloring scheduling". Cada examen es un vértice. Se conectan si comparten al menos un estudiante. Los colores son las franjas horarias. Se aplica también a: citas médicas, programación de procesos en computadores, asignación de canales de WiFi.

✓ Resumen de lo demostrado

  • • Terna {P}S{Q}: el greedy siempre produce un coloreo válido
  • • Invariantes I0–I3 se mantienen en toda iteración (inducción)
  • • Terminación: todos los bucles tienen función de cota finita
  • • Cota: colors-used ≤ Δ(G)+1 siempre garantizada

¿Por qué vale la pena verificar formalmente?

La verificación con invariantes da una certeza matemática: sin importar qué grafo le demos, el algoritmo se comporta como prometemos. Crucial cuando el software controla sistemas críticos como redes de telecomunicaciones o compiladores.
